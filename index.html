4.1 — Sublenguajes de SQL

DML
Data Manipulation Language

Si el DDL construye los cimientos de una base de datos, el DML es quien vive ahí todos los días: el lenguaje que inserta, lee, corrige y elimina los datos una vez que las tablas ya existen.

consola — base_de_datos
INSERTUPDATEDELETESELECT*

¿Qué es?

El lenguaje que manipula los datos

DML son las siglas de Data Manipulation Language (Lenguaje de Manipulación de Datos). Es el subconjunto de SQL encargado de trabajar con el contenido de las tablas: los registros, filas y valores que ya viven dentro de una estructura creada previamente con DDL.

Mientras el DDL responde a la pregunta "¿cómo se ve la base de datos?", el DML responde a "¿qué información hay dentro, y cómo la modifico?". No crea tablas ni esquemas: opera sobre las que ya existen.

En la práctica, casi todo lo que hace una aplicación contra su base de datos —guardar un registro de usuario, actualizar un saldo, borrar un pedido cancelado, mostrar una lista de productos— pasa por una instrucción DML.

Comandos principales

Las cuatro instrucciones base

El DML se apoya en cuatro verbos. Cada uno representa una acción distinta sobre los datos ya almacenados.

INSERT

Agrega nuevos registros a una tabla.

INSERT INTO alumnos (nombre, grupo) VALUES ('Ana López', '4°A');

SELECT

Consulta y recupera información existente.

SELECT nombre, grupo FROM alumnos WHERE grupo = '4°A';

UPDATE

Modifica valores en registros existentes.

UPDATE alumnos SET grupo = '4°B' WHERE nombre = 'Ana López';

DELETE

Elimina registros de una tabla.

DELETE FROM alumnos WHERE nombre = 'Ana López';
!

Nota técnica: Algunos catálogos académicos clasifican SELECT dentro de un sublenguaje aparte llamado DQL (Data Query Language), como se ve en la tabla del pizarrón. Otros lo incluyen directamente dentro de DML porque también "manipula" el resultado mostrado. Por eso lo verás mencionado en ambos grupos según el autor o la institución.

Características

Qué distingue al DML del resto

  • 01

    Opera sobre datos, no sobre estructura

    No crea, altera ni elimina tablas, columnas o bases de datos. Esa labor es del DDL. El DML solo toca el contenido que vive dentro de las estructuras ya definidas.

  • 02

    Es transaccional

    Sus cambios pueden agruparse en transacciones y revertirse con ROLLBACK o confirmarse con COMMIT antes de que sean permanentes, lo que da control y seguridad sobre cada modificación.

  • 03

    Usa la cláusula WHERE

    UPDATE y DELETE casi siempre se combinan con WHERE para limitar el alcance de la operación. Sin ella, el cambio afecta a todos los registros de la tabla.

  • 04

    Es el lenguaje más usado en el día a día

    Si el DDL se ejecuta pocas veces (al diseñar la base), el DML se ejecuta constantemente: cada clic de un usuario en una app suele traducirse en una instrucción DML.

  • 05

    Depende de permisos del DCL

    Quién puede insertar, actualizar o eliminar datos se controla con el Data Control Language (DCL), por lo que DML y DCL trabajan en conjunto dentro de un sistema seguro.

Usos comunes

Dónde se aplica en la vida real

Registro

Altas de información

Guardar un nuevo cliente, producto, pedido o cuenta de usuario al momento en que se crean.

Consulta

Reportes y listados

Mostrar inventarios, historiales de compra o resultados de búsqueda en una app o sitio web.

Edición

Actualización de estados

Cambiar el estatus de un pedido, corregir un dato erróneo o actualizar un saldo o inventario.

Limpieza

Eliminación de registros

Borrar cuentas inactivas, pedidos cancelados o datos de prueba que ya no se necesitan.

Migración

Carga de datos masiva

Insertar lotes de información al migrar de un sistema antiguo a uno nuevo.

Backend

Lógica de aplicaciones

Cada formulario, botón de "guardar" o "eliminar" en una app termina ejecutando DML detrás de cámaras.

Cómo se ejecuta

El ciclo de una operación DML

Una instrucción DML rara vez actúa sola. Normalmente forma parte de un ciclo que protege la integridad de los datos:

Iniciar transacción → Ejecutar INSERT/UPDATE/DELETE → Verificar resultado → COMMIT

Si algo falla en cualquier punto del proceso —por ejemplo, una conexión interrumpida o una validación que no se cumple— la transacción puede revertirse por completo con ROLLBACK, como si nunca hubiera ocurrido.

Sintaxis profunda

Cómo se escribe cada instrucción

Más allá del ejemplo básico, cada instrucción DML tiene variantes y cláusulas que amplían su potencia. Aquí la anatomía completa de cada una.

INSERT — agregar registros

SQL 
-- Insertar un solo registro
INSERT INTO productos (nombre, precio, stock)
VALUES ('Laptop Lenovo', 14999.00, 25);

-- Insertar múltiples filas a la vez
INSERT INTO productos (nombre, precio, stock)
VALUES
  ('Mouse Inalámbrico', 349.00, 100),
  ('Teclado Mecánico',  899.00, 50),
  ('Monitor 24"',       4200.00, 15);

-- Insertar desde otra tabla
INSERT INTO historial_precios (producto_id, precio, fecha)
SELECT id, precio, NOW() FROM productos;

SELECT — consultar datos

SQL 
-- Selección básica con filtro
SELECT nombre, precio
FROM productos
WHERE stock > 0
ORDER BY precio ASC;

-- Con funciones de agregación
SELECT
  COUNT(*) AS total_productos,
  AVG(precio) AS precio_promedio,
  SUM(stock) AS inventario_total
FROM productos
WHERE precio < 5000;

-- JOIN: combinar dos tablas
SELECT o.id, c.nombre, o.total
FROM ordenes o
JOIN clientes c ON o.cliente_id = c.id
WHERE o.estatus = 'pendiente';

UPDATE — modificar registros

SQL 
-- Actualizar un campo específico
UPDATE productos
SET precio = 12999.00
WHERE nombre = 'Laptop Lenovo';

-- Actualizar múltiples columnas
UPDATE clientes
SET
  correo   = 'nuevo@correo.mx',
  telefono = '555-9988',
  activo   = 1
WHERE id = 42;

-- Actualizar con cálculo (aplicar descuento del 10%)
UPDATE productos
SET precio = precio * 0.90
WHERE categoria = 'electrónica';

DELETE — eliminar registros

SQL 
-- Eliminar un registro por ID
DELETE FROM productos
WHERE id = 7;

-- Eliminar todos los registros de una condición
DELETE FROM sesiones
WHERE fecha_expiracion < NOW();

-- Con subconsulta: eliminar clientes sin órdenes
DELETE FROM clientes
WHERE id NOT IN (
  SELECT DISTINCT cliente_id FROM ordenes
);

Lo que no debes hacer

Errores comunes al usar DML

Conocer los errores más frecuentes puede ahorrarte horas de depuración y evitar pérdidas de datos irreversibles.

Crítico DELETE sin cláusula WHERE ▶

Omitir WHERE en un DELETE elimina todos los registros de la tabla sin confirmación. Es uno de los errores más costosos en producción.

SQL — MAL
-- ⚠ Borra TODA la tabla de clientes
DELETE FROM clientes;

-- ✔ Correcto: solo borra los inactivos
DELETE FROM clientes WHERE activo = 0;
Crítico UPDATE sin cláusula WHERE ▶

Al igual que DELETE, un UPDATE sin WHERE modifica cada fila de la tabla. Todos los salarios, estados o valores quedarán iguales.

SQL — MAL
-- ⚠ Cambia el depto. de TODOS los empleados
UPDATE empleados SET depto = 'Ventas';

-- ✔ Correcto: solo el empleado 5
UPDATE empleados SET depto = 'Ventas' WHERE id = 5;
Advertencia Insertar sin especificar columnas ▶

Si no se listan las columnas en un INSERT, el orden de los valores debe coincidir exactamente con el orden de la tabla. Cualquier cambio futuro en la estructura romperá la instrucción.

SQL
-- Frágil: depende del orden de columnas
INSERT INTO empleados VALUES (10, 'Pedro', 'TI', 18000, 1);

-- ✔ Robusto: columnas explícitas
INSERT INTO empleados (id, nombre, depto, salario, activo)
VALUES (10, 'Pedro', 'TI', 18000, 1);
Advertencia No usar transacciones en operaciones múltiples ▶

Cuando varias instrucciones DML dependen entre sí (por ejemplo, descontar saldo y registrar movimiento), ejecutarlas sin transacción puede dejar la BD en estado inconsistente si falla alguna.

SQL — Con transacción
START TRANSACTION;

UPDATE cuentas SET saldo = saldo - 500 WHERE id = 1;
UPDATE cuentas SET saldo = saldo + 500 WHERE id = 2;
INSERT INTO movimientos (origen, destino, monto) VALUES (1, 2, 500);

COMMIT; -- solo se guarda si todo salió bien
Buena práctica Olvidar cerrar comillas o escapar caracteres ▶

Los valores de texto con apóstrofes o comillas sin escapar rompen la sintaxis y pueden abrir vulnerabilidades de SQL Injection. Siempre usar parámetros preparados en aplicaciones.

SQL
-- ⚠ Error de sintaxis (apóstrofe sin escapar)
INSERT INTO clientes (nombre) VALUES ('O'Brien');

-- ✔ Escapado correcto en SQL
INSERT INTO clientes (nombre) VALUES ('O''Brien');

-- ✔ Mejor aún: parámetros preparados (en código)
-- INSERT INTO clientes (nombre) VALUES (?)

Referencia rápida

Cheat Sheet · DML

Un resumen de lo esencial para tener siempre a la mano.

INSERT

  • Agrega filas nuevas
  • Requiere columnas + valores
  • Puede insertar múltiples filas
  • Puede copiar desde otra tabla con SELECT

SELECT

  • Recupera datos sin modificarlos
  • Usa WHERE, ORDER BY, GROUP BY
  • Combina tablas con JOIN
  • Admite funciones: COUNT, SUM, AVG…

UPDATE

  • Modifica valores existentes
  • Siempre usar WHERE
  • Puede actualizar múltiples columnas
  • Admite cálculos en el SET

DELETE

  • Elimina filas (no la tabla)
  • Siempre usar WHERE
  • Se puede revertir con ROLLBACK
  • Más lento que TRUNCATE (DDL)
